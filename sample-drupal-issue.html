Overview

Similar to #3548395: Linker does not appear for certain field widgets, the linker does not appear. Discovered in #3554182: Add `type: array, items: { type: string }` test-only SDC, to prove "Tags" field can be linked, which itself was spawned by this Drupal CMS bug report: #3552918: Blog post tags don't appear in Canvas as potential field.


👆 no linker, despite suggestions being available — see #3554182-4: Add `type: array, items: { type: string }` test-only SDC, to prove "Tags" field can be linked.

Test by:

  1. instantiating a "Tags" test-only SDC provided by canvas_test_sdc since #3554182: Add `type: array, items: { type: string }` test-only SDC, to prove "Tags" field can be linked
  2. … on a content template for a node type that has an optional multiple-cardinality entity reference field, such as the typical field_tags field

Presumably, this is due to how it associates the linker with the field label, and multi-value field widgets appear to render the label differently.

⚠️ Note: this is arguably blocked on #3546869: Editing support for `type: array`: support multiple-cardinality Field Widgets (`StaticPropSource`s), because until that happens, technically multiple-cardinality field widgets are not supported at all by Canvas yet!

Manual testing

I tested the functionality, and it works using the tests/modules/canvas_test_sdc/components/tags/tags.component.yml component to link the field_tags on an article when installing in the standard core profile.

Issue review

The issue is not in a reviewable state.

  1. The proposed resolution in the summary is still "TBD," but from testing I can see it works, so that needs to be updated. I assume there are also other parts of the summary that needs to be updated
  2. There is no history of git commits; there is only 1 commit. Chandu7929, I assume this is because you force-pushed and wiped out all other commits except for the single commit. This is especially bad because I can tell from looking at the issue that there used to be a different approach that didn't solely rely on changes to the canvas_stark theme.

    For instance, I can tell from this commit, which is no longer in the branch history, that there was a solution that involved changes to src/Plugin/Canvas/ComponentSource/GeneratedFieldExplicitInputUxComponentSourceBase.php, which is no longer changed in the current MR. After this commit in #8, @narendrar commented, "This is how it looks now," and before that, in #5, bnjmnm set it to "needs review," indicating it worked. I can also see that Wimleers commented on the approach using GeneratedFieldExplicitInputUxComponentSourceBase.

    Also, there is no explanation as to why we are not using the previous approach. Since there is no git history, I cannot (easily) even check for myself what the previous approach was.

  3. There needs to be a test that proves this bug is fixed. Maybe the git history had a test, I can't tell

Next steps

@chandu7929 please:

  1. We need a new branch/MR that brings in the old approach, GeneratedFieldExplicitInputUxComponentSourceBase. Leave the current MR as is for comparison. There is no way to determine if your current approach is correct when I know there is a previous approach that also worked, and from what limited visibility I have into that approach, it is probably more correct. If you were able to recover the old git history, that would be best, but that might be lost now.

    Suggestions for how to do this:

    a) Use one of the tags that GitLab creates when you force push: https://git.drupalcode.org/issue/canvas-3554197/-/tags. I assume this will mean we have lost the git history 😥 (not ideal, but we can deal with that).
    b) Maybe someone has a local branch that has the git history and previous approach before it was wiped out by a force push.
    @chandu7929, if you don't have this, please check with @NarendraR, as he worked on it before you. To be clear, it is not NarendraR's or anyone else's responsibility to keep local branches as backups.

    We just should NOT be using force push to wipe out git history with 1 commit. If in the future you find this is the only solution, stop and contact another Canvas developer to find a better solution.

    If you create an MR via b), please confirm the changes match the latest git tag that used the previous approach found here: https://git.drupalcode.org/issue/canvas-3554197/-/tags. Please link to the latest tag in a comment here.
